One seam for ferry rules. All three callers cross the same interface. Ferry support in pathfinding unlocked without touching movement or track_builder.

Problem. Ferry crossing rules — detection, cost, reachability expansion, teleport, capacity limits — are fragmented across 4 modules. The include_ferries=False TODO in pathfinding.py cannot be resolved without reading 5+ functions across 3 other files.

Solution. Extract a FerryRules module whose interface is the complete set of ferry-related queries; every caller goes through it.

  • → locality: ferry bug fixes land in one module
  • → leverage: pathfinding ferry support becomes a 3-line change (call expand_reachable instead of inlining)
  • → interface shrinks: callers no longer import from hex_node for ferry details
  • → deletion test passes: deleting ferry calls from movement + track_builder + pathfinding exposes the same logic in all three — it was earning its keep

2. Deepen the build validator chain

Strong in-process
track_builder.py

Before — 8 shallow layers

check_node_valid ×2
check_adjacency
check_right_of_way
connectivity check ← computes reachable
check_milepost_limit
ferry capacity ← computes player set
check_city_access
check_blocking ← tier-1 only (TODO)

Each layer assumes the previous succeeded. Reachable set computed twice (connectivity + blocking). To test blocking, set up 7 other checks first.

After — one deep module

BuildValidationContext.build(game_state, player_id)

BuildValidator (deep module)

reachable_set — computed once, shared
player_track_map — computed once
validate(edge) → ValidationResult
execute_build uses result list atomically

Shared state pre-computed once. Each validator is a pure function accepting the context. Blocking rule test needs only a context with the right owned_edges — no 7-check setup.

Problem. validate_build_edge() chains 8 checks that are sequentially coupled — reachability is re-computed by both the connectivity check and the blocking check, and testing any one validator requires satisfying all the ones before it.

Solution. Extract a BuildValidator module that pre-computes shared state (reachable set, player track map) once per execute_build call, then exposes a validate(edge) that each check can call as a pure function.

  • → locality: reachability computation lives once; the tier-2 blocking TODO becomes a one-function edit
  • → leverage: each validator testable in isolation — pass a context with just the fields it needs
  • → interface shrinks: execute_build calls one validate; callers never touch the 8 sub-checks
  • → performance: reachable set O(V+E) computed once per build call, not twice

3. Introduce a commit/rollback seam on GameState

Worth exploring in-process
game_state.py · movement.py · track_builder.py

Before — mutation with partial failure

caller
execute_operate()
action 1 — mutates train.current_node ✓
action 2 — mutates player.ecu ✓
action 3 — validation fails ✗
state partially mutated — no rollback

Tests work around this by creating a fresh GameState per test. The prototype calls execute_operate speculatively without a safety net.

After — snapshot or staging seam

sequenceDiagram
  participant C as caller
  participant GS as GameState
  participant EO as execute_operate

  C->>GS: snapshot()
  C->>EO: execute(actions, gs_copy)
  EO-->>C: OperateResult(ok, new_state)
  C->>GS: commit(new_state)

  note over C,EO: failure: discard new_state, gs unchanged

Callers get atomicity. Tests never need fresh state setup — pass any snapshot. Prototype can speculatively evaluate moves.

Problem. execute_operate() and execute_build() mutate GameState in-place; a validation failure mid-sequence leaves state partially corrupted. The prototype and any future UI must defensively re-create state before any speculative evaluation.

Solution. Add a snapshot() method to GameState and make execute_operate/execute_build accept an optional copy, returning a new state on success. The seam sits between "validate + compute" and "commit."

  • → locality: rollback logic lives in one snapshot method, not scattered across callers
  • → leverage: speculative pathfinding ("what if I build here?") becomes a one-liner — snapshot, evaluate, discard
  • → interface shrinks: callers no longer need to track pre-state manually
Note: dataclass copy.deepcopy() on nested dicts/sets has a cost. Measure before committing — full snapshot may only be worth it once the prototype needs speculative evaluation.

Problem. include_ferries=False appears in all three pathfinding functions with TODO comments. One adapter means a hypothetical seam — the parameter adds interface complexity with zero current leverage.

Solution. Either implement ferry traversal (enabled by Candidate 1's FerryRules module) to make the seam real, or delete the parameter and add ferry support as a separate function when needed.

  • → interface shrinks: callers no longer pass a flag that does nothing
  • → leverage (if implemented): prototype's pathfinding visualization correctly handles Dublin/Belfast ferry crossings
  • → locality (if cut): no phantom code to explain to future readers
Note: Candidate 1 (FerryRules) is a prerequisite for Option A — implementing ferry traversal without it would re-fragment ferry logic.

Top recommendation

Candidate 1 — Ferry Rules module

Ferry is the only concept that actively blocks forward progress right now — include_ferries=False is a three-file stub and the pathfinding module cannot be completed without it. Concentrating all ferry rules behind one module (Candidate 1) also unlocks Candidate 4 for free and makes Candidate 2's BuildValidator easier to write, since _ferry_expanded_reachable moves out of track_builder.py. It has the highest leverage-per-change ratio of the four.
