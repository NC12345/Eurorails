Problem. cost_of_edge() uses only TERRAIN_BUILD_COST, WATER_SURCHARGE, and HexNode methods — all map knowledge — yet it lives in build rules, forcing the graph-algorithms module to import the build-rules module.

Solution. Move cost_of_edge() to hex_node.py; both track_builder and pathfinding import it from there.

  • → leverage: hex_node.py interface deepens — terrain + obstacle + cost behind one import
  • → wrong-direction dependency eliminated
  • → cost_of_edge() testable without build rules in scope

Make execute_operate() atomic

Strong

movement.py:25-82

Before — incremental apply

MoveTo(r5_c10) ✓ — state mutated
MoveTo(r6_c11) ✓ — state mutated
PickUp(Wheat) ✗ — no resource here
Return error. Train already moved.
GameState is now in partial apply.

After — snapshot + rollback

Snapshot state before sequence
Simulate MoveTo(r5_c10)
Simulate MoveTo(r6_c11)
PickUp(Wheat) ✗ — no resource here
Restore snapshot. Return error.
GameState unchanged — zero partial apply.

Problem. execute_operate() applies each action immediately; a failure mid-sequence returns an error but leaves train position, cargo, ECU, and resource supply in a partially-applied state — with no way to recover. execute_build() is already atomic.

Solution. Before the action loop, snapshot mutable fields (TrainState, PlayerState.ecu, cargo, resource_supply, hand); restore on first error.

  • → locality: failure handling concentrated at one point, not scattered across 6 action handlers
  • → interface contract matches execute_build() — callers get consistent all-or-nothing semantics
  • → real bugs in incremental test coverage become visible

Extract traversability predicate from pathfinding

Worth exploring

Adapters

Problem. _movement_traversable() hardcodes the major-city-interior game rule inside pathfinding. Pathfinding should be a map-generic algorithm; the traversability policy is a caller concern.

Solution. Accept a traversable: Callable[[str, str], bool] predicate. Callers supply the rule; pathfinding stays pure.

  • → leverage: one seam covers all future traversability variants (ferries, city type changes)
  • → pathfinding testable with lambda a, b: True — no game-state setup needed
  • → one adapter per context, not one hard-coded rule

One adapter exists today (major-city interior). Seam becomes real when ferry traversal needs a different rule — currently that's stubbed. Worth noting but not urgent.

Unify map loading in visualization

Worth exploring

used by: movement · track_builder · pathfinding · tests

After — one seam

Problem. master_map_visualization.py opens master_map.json directly and accesses raw dict keys instead of calling game_state.load_map() — giving it no access to HexNode predicates and creating a second loading seam that must track schema changes independently.

Solution. Replace the open() call with load_map(); update field accesses to use HexNode attributes.

  • → locality: single loading seam for the entire codebase
  • → visualization inherits all future HexNode methods automatically

Top recommendation

Make execute_operate() atomic

The non-atomic operate phase is the only place in the rules engine that can leave GameState in an inconsistent condition. execute_build() already proves the staging pattern works here. Fixing this closes a real correctness gap — not just an aesthetic one — and the test suite has no coverage for the partial-apply failure modes it currently allows. Tackle this first, then candidate 1 (cost_of_edge placement) as a low-risk, high-clarity cleanup.
